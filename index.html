Introduction to Website Deployment with GitHub webpages.

We have several ways to deploy a website to the public internet.
To explain how a website deployment works we will be using the GitHub webpages.

First that all lets start with what GitHub is....

GitHub is a web-based Git or version control repository and Internet hosting service. It offers all of the distributed version control and source code management (SCM) functionality of Git as well as adding its own features. It provides access control and several collaboration features such as bug tracking, feature requests, task management, and wikis for every project

Considering the above mentioned a GitHub Pages is a static site hosting service. That's mean that a Github page is designed to host your personal, organization, or project pages directly from a GitHub repository.

Now that we have the account & the Repository is important to know that we have two ways to upload our site to Github. The first is directly on the GitHub pages (by Front End ) and the second is by using the command lines (Back end)

On the personal I prefer the Front end deployment option . The same due to it´s the most practical way to apply it. However is good to know both options in case that you could face any issue on the website. So... lets explore the backend option:

I.- Back end option to upload a site on GitHub (This option is generally used by ruby and jekyll generation files use)

By using Git we will push (upload) the contents of the site´s directory to the new repository.
To do so, we´ll initialize a Git repository in your sites directory.

  • Open the terminal and use cd command to navigate to the sites directory.
  • Initialize a Git repository with the git init command.
  • Specify to Git where the repository will store your content site(The one you created on GitHub earlier). In order to specify the repository using Git is required to add the remote and label it as the origin.
  • The remote is the URL of the repository that will store the site´s contents.
  • The Origin is an alias for the remote. The before mention is in order to avoide to having all the time type the lengthy remote URL over and over again, you can simply refer to it as origin later on.

In the terminal , you can add the remote with the next command:

git remote add origin https://github.com/your-user-name/your-user-name.github.io.git

the URL https://github.com/your-user-name/your-user-name.github.io.git Is the remote URL that refers to the repository we have created on GitHub earlier.
⦁ In the terminal, add the remote that points to the repository we created earlier

Example:
git remote rm origin

After specify the remote we will check that it has been added properly with the next command: git remote -v (This command will list all the Git remotes and their corresponding URLS).
After complete the mentioned steps is required to know exactly which files should be pushed to your repository.
In case we would like to push all the site´s content to the repository . We will do the following thing:

  1. Add all your site content to the staging area by using the git add command.
  2. Save your changes by usin the commit -m "Save my work" command.

Great, Now lets go to deploy our site! Starting deployment by typing the git push -u origin master command. Important: Remember that on this deploy step the authentification credentials will be requested.


II.- Front end option to upload a site on GitHub (This option is generally used when we create files by using HTML / CSS /JS languages and editors like Notepad++ or Atoms).

Go to your GitHub webpage and access your repository github.io.
Once you are here you can start by uploading or creating files on your master branch.
Note I: Everything we start on our master branch and that we'll commmit will be reflected on our URL. Due to the same this branch is consider like the main of the branches.
Considering the above mentioned is recommended to use gh-pages branch to apply changes or new versions over those to avoid affecting the master branch and also to be able to apply proper rollbacks.
After commit our master branch we will be automatically publishing our site changes on the public Internet.


Note II: For the new changes worked on the gh-pages branches we must apply a pull request after commit our changes. These pull request will do two things, the first is to compare the new changes with the old version and the second thing is complete the a proper pull request authroization process to validate that new changes wont affect previous versions.
